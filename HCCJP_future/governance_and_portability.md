## スライド素材

### ペット vs コンテナ — AI時代に同じ問いが来た

| | 昔のインフラ | AI エージェント時代 |
|--|------------|-----------------|
| 🐕 **ペット** | 長年育てた手作りVM | ローカル環境（MCP・認証・設定が積み上がる） |
| 🐄 **コンテナ** | 使い捨て・宣言的に再現 | クラウドVM（ephemeral・GitHub経由のみ） |

インフラ界が「ペット→コンテナ」を答えとして出した

**しかしAIエージェントには、同じ答えが通じない部分がある**
→ なぜか？

---

### AIが解いたこと: 「魂のポータビリティ」

#### 昔の「ペットVM」の本質的な問題

「ひれのタレ」= 設定・認証・スクリプト・ノウハウがVMの中に積み上がる
→ 移せない。再現できない。消えたら終わり。

#### AIエージェント時代の革命

```
CLAUDE.md / skills / rules / context = すべてテキスト
  → Git で管理できる
  → どこでも clone して引き出せる
  → 実行環境は cattle でいい
```

**「魂（指示・哲学）」と「肉体（実行環境）」を初めて分離できた**

従来のソフトウェアにはできなかった分離。AIエージェントだから実現した。

---

### でも、残る問い: 認証・権限・ガバナンス

「魂」はポータブルになった。でも「肉体」の選択には答えがいる。

| 実行環境 | 自由度 | 扱える情報 |
|---------|--------|-----------|
| 純粋個人ローカル | ◎ | MCP・全API・全認証情報 |
| 企業PCのローカル | △ | 条件付きアクセス・MDM制約内 |
| クラウドVM（ephemeral） | △ | GitHub経由の情報のみ |
| エージェントへの権限委任 | ❓ | **どこまで渡す？誰が責任を持つ？** |

→ どのコンテキストを、どの環境で、どこまで扱わせるか
**これがこれからのインフラ・ガバナンス設計の核心**

---

### 個人 → チーム → エンタープライズで答えが変わる

| スコープ | コンテキスト管理 | ガバナンス | 主な課題 |
|---------|--------------|----------|---------|
| **個人** | CLAUDE.md 1ファイル | 自分だけ | 自由だが属人化 |
| **チーム** | 共有リポジトリ | 誰がエージェントを信頼するか | 権限設計 |
| **エンタープライズ** | ポリシー配下 | M365・条件付きアクセス | コンプライアンス vs 柔軟性 |

#### まだ答えは出ていない

- **AIエージェント時代は始まったばかり**
- どこで動かすか × 何を渡すか × 誰が責任を持つか
- この3つが、これからのインフラ・ガバナンス設計の問いになる

---

### でも本質は「軽さ」にある

#### AI指示 = テキスト = 究極に軽い

- CLAUDE.md・skills・MCP設定 → Git で管理・どこでも再現
- 同じ指示で、**どこでも同じことができる**
- ポータビリティは既に実現されている

#### なのに、なぜ「どこでも動けない」のか？

→ **ガバナンスが「デバイス」に縛られているから**

---

### 今のギャップ — デバイス依存のガバナンス

```
個人スマホ → AIエージェント → 会社データ  ❌ アクセス不可
会社PC     → AIエージェント → 会社データ  ✅ アクセス可

同じ指示・同じAI・同じ能力
  でも「どのデバイスか」で結果が変わる
```

**現実の迂回策**: 会社PCにSSHで入り、そこで実行
→ 「無駄なワンホップが挟まるだけで、何も守られていない」

ガバナンスの境界がデバイスである限り、
AIの軽さとガバナンスの重さの矛盾は解消されない

---

### これからの姿 — エージェントへの条件付きアクセス

#### Zero Trust の進化

| 世代 | 信頼の単位 | 判断基準 |
|-----|-----------|---------|
| Zero Trust 1.0 | ネットワーク → **デバイス** | このデバイスは信頼できるか |
| Zero Trust 2.0 | デバイス → **アイデンティティ** | このユーザーは信頼できるか |
| **AI Agent 時代** | アイデンティティ → **行動** | このエージェントの、この行動は、今許可してよいか |

- 「会社PCだから許可」ではなく「このエージェントがこの操作を今行うリスクは？」
- リスクがあれば止める。問題なければどこでも動く
- **デバイスではなく、コンテキストと行動でガバナンス**

### AIエージェント前提で考えると

- **コンテキストの設計**が最重要
  （どこに何を持たせるか,これは代替えが効かない）
- **モデル, エージェントソフトウェア選択の柔軟性**を担保する
  （ベンダーロックイン回避）
- **実行場所を固定しない**
  （ローカル / クラウド / ハイブリッド）
- **セキュリティ境界**の再定義
  （エージェントがAPIを叩く世界のガバナンス）

---

### AI エージェントの4層モデル

```
┌─────────────────────────────────────────────────┐
│ 📜 コンテキスト（魂）                            │ ← 自分で作る・育てる
│    CLAUDE.md / skills / rules / MCP設定          │   （組織の資産）
├─────────────────────────────────────────────────┤
│ 🤖 エージェントSW                                │ ← 選ぶ・使う
│    Claude Code / Copilot / Codex / Gemini CLI    │   （ほぼクローズドソース）
├─────────────────────────────────────────────────┤
│ 🧠 頭脳（モデル）                                │ ← 借りる・差し替える
│    Claude / GPT-4o / DeepSeek / OSS LLM          │   （最強を選ぶ）
├─────────────────────────────────────────────────┤
│ 🏃 実行環境（肉体）                              │ ← インフラとして整備
│    ローカル / クラウドVM / エッジ                │
└─────────────────────────────────────────────────┘
```

**この4層それぞれに、別々の「誰が作るか」がある**

---

### 同じ頭脳でも、体験が全然違う

実体験：Claude を使っているのに

- **Copilot + Claude** → 期待通りに動かない 😞
- **Claude Code + Claude** → 圧倒的に期待通りに動く 🚀

**頭脳は同じ。実行環境も同じ。違うのはエージェントSWだけ。**

→ エージェントSW層の品質が、体験の大半を決めている

---

### エージェントSW層が担うこと

#### RAG時代 → エージェント時代の進化

| 時代 | アプローチ | 限界 |
|------|-----------|------|
| **RAG時代** | ベクターDB → 文書検索 → LLMに丸投げ | 「関連文書」が答えとは限らない |
| **エージェント時代** | 今何をすべきか自分で考え、ツールを選び、連鎖させる | エージェントSWの実装品質に依存 |

エージェントSWが担うこと：
- どのツールをいつ使うか（Tool selection）
- 失敗したらどうリトライするか（Retry logic）
- コンテキストをどう圧縮・管理するか（Context management）
- 複数エージェントをどう協調させるか（Orchestration）

**この「ループの実装品質」が体験の差になる**

---

### エージェントSWの現実: ほぼクローズドソース

| ツール | 提供元 | ソース | 自作手段 |
|--------|--------|--------|---------|
| Claude Code | Anthropic | ❌ 非公開 | SDK / API |
| GitHub Copilot | Microsoft | ❌ 非公開 | Extensions |
| Codex CLI | OpenAI | ❌ 非公開 | API |
| Gemini CLI | Google | △ 一部公開 | API |

- SDKを使えば企業が自作することも**技術的には可能**
- でも… **本当に自作すべきか？**

---

### 企業はどこに注力すべきか

| 層 | 誰が作るか | 企業の判断 |
|----|-----------|-----------|
| 📜 **コンテキスト** | **自分たちで作る・育てる** | ✅ ここが組織の競争優位 |
| 🤖 **エージェントSW** | 現時点はClaude Codeが頭1つ飛び抜け | ⚡ 自作より使う方が圧倒的に合理的 |
| 🧠 **モデル** | 外部サービスを借りる | 🔄 最強・最コスパに随時切り替え |
| 🏃 **実行環境** | インフラチームが整備 | 🏗️ APIが使えれば何でもいい |

---

### コンテキストは「有限の高級資源」

エージェントへのブリーフィング

優秀なアシスタントに依頼するとき:
- ❌ 関係ない話を延々と前置きする
- ❌ 古い情報・ノイズを大量に渡す
- ✅ 今のタスクに必要な情報だけを的確に伝える

**AIエージェントも同じ。**
コンテキストの**量**より**質**が、性能を決める。

---

### コンテキスト汚染が起きると

```
コンテキスト = [必要な情報] + [古い議論] + [無関係なログ] + [重複定義]
                    ↓
  エージェントが「どれが今重要か」を判断するコストが増える
  本当に重要な指示が薄まる
  期待と違う行動をとる
  同じことを何度も間違える
```

**「余計なものを入れない」はハードスキル**

---

### AI エージェント時代の新しいリテラシー

| 旧時代のリテラシー | AI エージェント時代のリテラシー |
|----------------|--------------------------|
| 検索ワードの選び方 | **コンテキストの設計と取捨選択** |
| コードが読める | **エージェントへの指示が書ける** |
| ツールを使いこなす | **ツールをエージェントに使わせる** |
| 情報を記憶する | **情報をどこに置くかを設計する** |

---

### 最強の頭脳は、自分では持てない

- 世界最強の計算資源は **一握りの組織に集中**している
- その組織がサービスを提供する限り、**誰でも最強の頭脳を使える**
- しかし自分たちで丸ごと持つことは（今は）できない

→ 外部サービスとして「借りる」ことで初めて使える頭脳

**それなら、いつでも最強に差し替えられる設計こそが合理的**

---

### 特定モデルへのロックインは組織の自殺行為

| 状態 | 結果 |
|------|------|
| 「うちはAモデルしか使えない」 | 競合がBモデルに乗り換えた瞬間に遅れを取る |
| 「うちはこのクラウドのAIしか許可していない」 | 最強モデルが別クラウドに出た瞬間に詰む |
| **その時々の最強（最コスパ）を自由に切り替え** | **常に最前線に立てる** |

ガバナンスは**モデル制限**ではなく**行動制限**で実現すべき

- どのモデルを使うか → 自由
- そのモデルが何をするか → ガバナンスで制御

---

### 深読み: なぜ引き継ぎは一方向なのか？（Part 5 からカット）

#### Web → ローカル（teleport）は「昇格」
- クラウドの成果を、より能力の高い環境に持ち込む
- MCP・ローカルファイル・認証情報が**増える**方向

#### ローカル → Web は「降格」（現時点で非対応）
- MCP、ローカルファイル、認証情報が**失われる**
- 技術的には実装可能だが、あまり自然なユースケースがないため優先度低い

---

### 設計思想（Part 5 からカット）

```
ローカル = sovereign environment（主権環境）= あなたの「城」
クラウドVM = task execution environment = 「出張先の作業場」
```

- **teleport** = 出張の成果を城に持ち帰る ✅ 自然
- **--remote** = 出張先に新しい仕事を投げる ✅ 自然（新規セッション）
- **Remote Control** = 城を動かさず、窓だけ外に出す ✅ 自然

→ 一方向性は技術的制約ではなく**設計哲学**
